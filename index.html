1. Intro

Hello readers,

In this blog, we will talk about one of the most important ideas in modern backend engineering: the event loop.

The event loop is one of those concepts that sounds complicated at first, but once you understand it from first principles, it becomes very natural.

More importantly, we will also understand how the event loop works together with connection pools.

This is where the concept becomes really useful in real backend systems.

We will answer questions like:

How can one event loop handle many database connections?

If the event loop is single-threaded, how can multiple queries be in progress at the same time?

What exactly is waiting?

Who is doing the actual work?

What happens when all connections in a pool are busy?

How does a waiting request resume later?

Let us build the idea slowly.

2. Event Loop From First Principles

Imagine you are running a small samosa shop.

There is only one person at the counter: you.

Customers keep coming in.

Some customers just want one ready-made samosa. Easy.

Some customers order something that takes time, like a fresh batch of samosas that needs to be fried.

Now imagine this:

A customer asks for fresh samosas, and you personally stand near the kadhai waiting for them to fry.

During that time, every other customer has to wait.

That would be slow.

A better way is:

  1. Take the fresh samosa order.
  2. Ask the cook to fry the samosas.
  3. Tell the cook, "Call me when they are ready."
  4. Go back to the counter.
  5. Serve the next customer.
  6. When the cook says the samosas are ready, come back and finish that order.

This is the basic idea behind the event loop.

The event loop is not about doing everything faster.

It is about not wasting time while something else is waiting.

At a very simple level, you can imagine the event loop as a loop that keeps asking:

Is there any work ready to run?

Did any timer complete?

Did any network response arrive?

Did any socket become readable?

Did any socket become writable?

Did any database query finish?

Is any callback, promise, or continuation ready to continue?

If something is ready, it runs that piece of work.

If something is not ready, it does not block the entire program waiting for it.

Conceptually: 

while (true) {
  checkTimers();
  checkNetworkEvents();
  checkFileSystemEvents();
  checkDatabaseSocketEvents();
  runReadyCallbacks();
}

This is not exactly how production runtimes implement it, but it is a useful mental model.

In real runtimes, the event loop does not blindly waste CPU by continuously checking everything.

Instead, it asks the operating system:

Wake me up when one of these sockets, timers, or file operations becomes ready.

This distinction is important.

The event loop is not usually busy-spinning.

It sleeps efficiently until the operating system reports that something is ready.

3. The Problem: Waiting Is Expensive

A program usually does two kinds of work.

3.1. CPU Work

CPU work means the program is actively using the CPU.

Examples:

  • Parsing JSON.
  • Sorting an array.
  • Compressing data.
  • Running business logic.
  • Rendering templates.
  • Encrypting or hashing data.

This kind of work needs CPU cycles.

If the CPU is busy doing this work, it cannot do something else at the same time on the same thread.

3.2. I/O Work

I/O means input/output.

Examples:

  • Calling another API.
  • Reading a file.
  • Querying a database.
  • Waiting for Redis.
  • Sending data over a socket.
  • Reading data from Kafka.
  • Writing logs to disk.

I/O work is different from CPU work because most of the time is spent waiting.

For example, your application sends this query to a database: 

SELECT * FROM users WHERE id = '1234';

After sending the query, your application is mostly waiting.

The actual query execution happens somewhere else:

  • Inside the database server.
  • On the database machine.
  • Using database CPU, memory, indexes, locks, buffers, and query planner.

Your application does not need to burn CPU while the database is working.

It only needs to be notified when the result comes back.

That is exactly where the event loop helps.

4. The Naive Way: Blocking Execution

Imagine a server handling one request.

Request 1 comes in.

Server:

  1. Read request.
  2. Send database query.
  3. Wait.
  4. Wait.
  5. Wait.
  6. Database responds.
  7. Build response.
  8. Send response.

This works fine for one request.

But what happens when many requests come in?

Request 1 -> waiting for database

Request 2 -> waiting for database

Request 3 -> waiting for Redis

Request 4 -> waiting for another HTTP API

If every request blocks a thread while waiting, the server needs many threads.

Threads are useful, but they are not free.

Each thread has costs:

  • Memory for stack.
  • Context switching overhead.
  • Scheduler overhead.
  • Synchronization complexity.
  • More chances of lock contention.
  • More pressure on the operating system.

So the question becomes:

Can we handle many waiting tasks without creating one blocked thread per task?

This is where event-loop-based systems are powerful.

5. The Core Idea of the Event Loop

The event loop follows a simple rule:

Do not wait doing nothing.

Start the slow operation.

Register what should happen when it finishes.

Move on to other work.

Come back when the result is ready.

In other words, the program does not block the main execution thread for slow I/O.

Instead, it says:

When this operation completes, put its callback or continuation into a queue.

The event loop will run it later.

This is why event loops are very good for I/O-heavy systems.

They allow one thread to coordinate many pending operations.

But here is the key point:

The event loop is not doing all the I/O work by itself.

It is coordinating I/O readiness.

The actual waiting happens in the operating system, network stack, database server, disk subsystem, or external service.

6. The Main Characters

To understand the event loop properly, we need to understand a few main components.

6.1. Call Stack

The call stack is where currently running code lives.

If one function calls another function, that function is pushed onto the stack.

JavaScript example: 

function a() {
  b();
}

function b() {
  c();
}

function c() {
  console.log("hello");
}

a();

The call stack roughly looks like this: 

c()
b()
a()

When c finishes, it is removed.

Then b finishes.

Then a finishes.

The call stack handles synchronous execution.

If the call stack is busy running a long computation, the event loop cannot run another callback on that same thread.

6.2. System APIs / Runtime APIs

Async operations are usually handed off to the runtime or operating system.

Examples:

  • Timers.
  • TCP sockets.
  • File system operations.
  • DNS lookups.
  • Database driver operations.
  • HTTP client operations.

For example: 

setTimeout(() => {
  console.log("timer done");
}, 1000);

The JavaScript engine does not sit and count one second manually.

The timer is registered with the runtime.

Later, when the timer expires, the callback becomes eligible to run.

6.3. Callback Queue / Task Queue

When an async operation completes, its callback or continuation is placed into a queue.

Examples:

Timer completed -> callback goes into task queue.

HTTP response arrived -> callback goes into task queue.

Database socket received data -> query continuation becomes ready.

File read completed -> callback goes into task queue.

6.4. Event Loop

The event loop coordinates all of this.

It repeatedly does something like:

Is the call stack empty?

Is there any completed task ready to run?

If yes, put that task onto the call stack and execute it.

This is the heart of the model.

7. Simple Event Loop Example

Consider this JavaScript program: 

console.log("A");

setTimeout(() => {
  console.log("B");
}, 1000);

console.log("C");

The output is: 

A
C
B

Why?

Because setTimeout does not block the main flow.

The flow is:

  1. Print A.
  2. Register timer.
  3. Continue immediately.
  4. Print C.
  5. Timer completes after one second.
  6. Callback becomes ready.
  7. Event loop runs callback.
  8. Print B.

The important idea is:

The waiting happened outside the main call stack.

8. Event Loop in Server Applications

Now let us move from a small timer example to a backend server.

Imagine a server receives three requests.

Request A: Get user profile.

Request B: Get order history.

Request C: Get payment status.

Each request needs a database query.

With event-loop-style handling, the server can do this:

Request A arrives.

Start database query for A.

Do not block.

Request B arrives.

Start database query for B.

Do not block.

Request C arrives.

Start database query for C.

Do not block.

Database result for B arrives first.

Continue Request B.

Database result for A arrives later.

Continue Request A.

Database result for C arrives after that.

Continue Request C.

Notice something important.

The responses do not have to come back in the same order as the requests.

The event loop handles whichever operation becomes ready first.

9. Important Point: Event Loop Does Not Make CPU Work Free

The event loop is excellent for I/O-heavy work.

But if you run heavy CPU work on the event loop thread, it blocks everything else.

Example: 

while (true) {
  // infinite loop
}

If this runs on the main event loop thread, nothing else can continue.

The samosa shopkeeper is now stuck doing one huge task and cannot return to the counter.

So the event loop should stay light.

For CPU-heavy work, systems usually use:

  • Worker threads.
  • Background jobs.
  • Separate processes.
  • Queues.
  • Batch workers.
  • Native extensions.
  • Dedicated compute services.

The event loop is best used as a coordinator, not as a place for expensive computation.

10. Connection Pools

Now let us connect the event loop to connection pools.

A connection pool is a group of already-created connections that the application reuses.

These connections are usually for:

  • Databases.
  • Redis.
  • Kafka.
  • HTTP services.
  • Message brokers.

Creating a new connection for every request is expensive.

For a database connection, the application may need to:

  1. Open a TCP connection.
  2. Perform TLS negotiation.
  3. Authenticate.
  4. Allocate server-side resources.
  5. Initialize connection state.
  6. Possibly set session variables.
  7. Possibly prepare statements or protocol state.

Doing this for every request would be slow.

So applications create a limited number of connections and reuse them.

11. The Important Question

Now we reach the main question:

If one event loop is single-threaded, how can it handle multiple database connections at the same time?

The answer is:

Because the event loop is not executing all database queries itself.

Each database connection is backed by a network socket.

The event loop registers interest in socket events.

The operating system watches those sockets.

When any socket becomes readable or writable, the OS notifies the runtime.

The event loop then runs the callback/continuation associated with that socket.

So one event loop can manage many connections because it is mostly waiting for readiness notifications.

It is not blocking on each connection one by one.

12. Database Connections Are Usually Sockets

A database connection is usually built on top of a TCP socket.

For example, when your application connects to PostgreSQL, MySQL, Redis, or another database, there is usually a network socket underneath.

A socket can have events like:

This socket is ready to write.

This socket has data available to read.

This socket was closed.

This socket had an error.

This socket timed out.

The operating system is very good at monitoring many sockets.

Modern operating systems provide mechanisms for this, such as:

  • epoll on Linux.
  • kqueue on BSD/macOS.
  • IOCP on Windows.
  • Other event notification mechanisms depending on the platform.

The runtime uses these mechanisms internally.

So instead of having one thread blocked on every connection, the runtime can say:

Here are 500 sockets.

Tell me when any of them becomes ready.

Then the thread can sleep.

When something happens on any socket, the OS wakes it up.

The event loop then processes the ready events.

13. One Event Loop, Multiple Connections

Suppose your application has a database pool with three connections. 

Pool size = 3

conn-1 -> socket-1
conn-2 -> socket-2
conn-3 -> socket-3

Now three requests arrive: 

Request A
Request B
Request C

The pool gives one connection to each request: 

Request A uses conn-1
Request B uses conn-2
Request C uses conn-3

Each request sends a query.

The important thing is that sending the query is usually fast.

The application writes bytes to the socket: 

conn-1 socket -> send query A
conn-2 socket -> send query B
conn-3 socket -> send query C

After sending the query, the application does not need to wait synchronously.

The queries are now in progress.

But where are they in progress?

Inside the database server.

On the network.

Inside OS buffers.

Inside the database execution engine.

The event loop is free to do other work.

Later, the database replies.

Maybe query B finishes first.

The OS notices:

socket-2 has data ready to read.

The event loop wakes up and says:

The socket for conn-2 is readable.

Run the callback associated with conn-2.

Parse the database response.

Resolve the promise for Request B.

Continue Request B.

Then maybe query A finishes.

socket-1 has data ready to read.

The event loop handles Request A.

Then query C finishes.

socket-3 has data ready to read.

The event loop handles Request C.

This is how one event loop handles multiple connections.

It is not doing this: 

Wait for conn-1.
Then wait for conn-2.
Then wait for conn-3.

It is doing this: 

Register interest in conn-1, conn-2, conn-3.

Sleep until any connection has something ready.

Handle whichever one becomes ready first.

That is the key difference.

14. Connection Pool State

A connection pool usually tracks three groups of things. 

Pool
├── available connections
│   ├── conn-1
│   └── conn-2
├── busy connections
│   ├── conn-3
│   └── conn-4
└── waiting queue
    ├── request-17
    ├── request-18
    └── request-19

The pool has to answer questions like:

Is there a free connection?

Is this connection currently in use?

Which request is waiting next?

Did this connection fail?

Should we create another connection?

Should we close an idle connection?

Did a request wait too long?

Should this connection be recycled?

This is mostly bookkeeping.

But the bookkeeping is driven by events.

Examples:

New request wants a connection.

Connection became free.

Database response arrived.

Connection errored.

Connection timed out.

Waiting request timed out.

Idle connection expired.

The event loop is what lets the application react to these events without blocking.

15. Acquire: What Happens When a Request Needs a Connection?

Suppose the pool size is 2. 

Pool size = 2

conn-1: free
conn-2: free

Now three requests arrive almost together. 

Request A
Request B
Request C

Request A asks for a connection. 

Pool gives conn-1 to Request A.

Request B asks for a connection. 

Pool gives conn-2 to Request B.

Request C asks for a connection. 

No connection is free.

Pool puts Request C into waiting queue.

The state is now: 

conn-1: busy with Request A
conn-2: busy with Request B

waiting queue:
- Request C

The important part:

Request C is waiting, but the event loop is not blocked.

Request C is represented by something like:

  • A promise.
  • A callback.
  • A deferred value.
  • A continuation.
  • A parked fiber/coroutine depending on the runtime.

The server can still handle other requests.

16. Release: What Happens When a Query Finishes?

Now suppose Request B's query finishes first.

The database sends response bytes on the socket for conn-2.

The OS marks that socket as readable.

The event loop receives the event.

The database driver reads the response, parses it, and resolves the query promise.

Request B continues.

Then Request B releases conn-2 back to the pool.

Now the pool checks:

Is anyone waiting for a connection?

Yes.

Request C is waiting.

So instead of putting conn-2 into the available list, the pool gives conn-2 directly to Request C.

The state becomes: 

conn-1: still busy with Request A
conn-2: now busy with Request C

waiting queue:
empty

This is the important flow:

A connection becomes free.

The pool resolves the next waiting acquire operation.

The waiting request becomes ready to continue.

The event loop later runs that continuation.

17. Query Flow With Pool

Consider this code: 

async function getUser(userId) {
  const conn = await pool.acquire();

  try {
    const user = await conn.query(
      "select * from users where id = ?",
      [userId]
    );

    return user;
  } finally {
    pool.release(conn);
  }
}

This looks simple, but there are multiple async boundaries.

17.1. Step 1: Acquire Connection

const conn = await pool.acquire();

If a connection is available, this may resolve quickly.

If no connection is available, this request is suspended.

But only this async function is suspended.

The event loop is not suspended.

The server can still process:

  • Other incoming requests.
  • Completed database responses.
  • Timers.
  • Socket events.
  • Other promise continuations.

17.2. Step 2: Send Query

const user = await conn.query(
  "select * from users where id = ?",
  [userId]
);

The database driver serializes the query into the database protocol.

Then it writes bytes to the connection socket.

Conceptually: 

Application -> socket -> network -> database server

After the query is sent, the application waits for a response.

But again, it does not block the event loop.

The driver registers that this query is waiting for data from this connection.

17.3. Step 3: Database Works Independently

The database server receives the query.

It may:

  • Parse SQL.
  • Plan the query.
  • Check permissions.
  • Use indexes.
  • Read pages from memory or disk.
  • Acquire locks.
  • Execute joins.
  • Build result rows.
  • Send response bytes back.

This work is not done by the application event loop.

It happens inside the database system.

17.4. Step 4: Socket Becomes Readable

When the database sends the result back, response bytes arrive on the application's socket.

The operating system notices:

This socket has data available.

The event loop receives this readiness event.

Then the database driver reads the bytes, parses the database protocol, and resolves the query promise.

17.5. Step 5: Function Continues

Now this line completes: 

const user = await conn.query(...);

The function continues and returns the user.

Then the finally block runs. 

pool.release(conn);

The connection goes back to the pool, or it is immediately assigned to another waiting request.

18. Important Distinction: Concurrency vs Parallelism

This topic often causes confusion.

18.1. Concurrency

Concurrency means multiple tasks are in progress during the same time period.

Example:

Request A is waiting for DB.

Request B is waiting for Redis.

Request C is waiting for another API.

Request D is being parsed.

Request E is ready to respond.

They are all in progress.

But not necessarily executing CPU instructions at the exact same instant on the same thread.

18.2. Parallelism

Parallelism means multiple tasks are literally executing at the same time, usually on multiple CPU cores.

Example:

Thread 1 is running task A on CPU core 1.

Thread 2 is running task B on CPU core 2.

Thread 3 is running task C on CPU core 3.

An event loop gives excellent concurrency for I/O-heavy work.

It does not automatically give CPU parallelism.

For CPU parallelism, you need multiple threads, processes, workers, or machines.

19. Why One Event Loop Can Handle Many Connections

One event loop can handle many connections because most connections are idle most of the time.

A database connection usually alternates between states: 

idle in pool
borrowed by request
query sent
waiting for response
response readable
processing response briefly
released to pool

The actual CPU time needed in the application for each connection is often small.

Most of the time is spent waiting for:

  • Network latency.
  • Database execution.
  • Locks.
  • Disk reads.
  • Remote service response.
  • OS buffers.

The event loop only needs to wake up when something interesting happens.

That is why a single event loop can coordinate thousands of sockets in some systems.

But this does not mean unlimited capacity.

There are still limits:

  • CPU time for callbacks.
  • Memory usage.
  • Socket/file descriptor limits.
  • Database capacity.
  • Pool size.
  • Network bandwidth.
  • Query complexity.
  • Garbage collection overhead.
  • Application-level bottlenecks.

20. Why Not Make the Pool Very Large?

At first, it may seem like a bigger pool is always better.

If 10 connections are good, maybe 1000 connections are better?

Usually, this is wrong.

Every connection consumes resources.

On the application side:

  • Memory.
  • Socket/file descriptor.
  • TLS state.
  • Buffers.
  • Pool bookkeeping.
  • Query state.

On the database side:

  • Memory.
  • Backend process or worker.
  • Connection state.
  • Locks.
  • Transaction state.
  • Query execution resources.
  • Cache pressure.

If the pool is too small, requests wait too much.

If the pool is too large, the database can become overloaded.

A connection pool is not just a performance optimization.

It is also a backpressure mechanism.

It says:

Only this many database operations can actively use connections at once.

Everyone else must wait.

This protects the database from being flooded.

21. Pool Waiting Queue Is Backpressure

When all connections are busy, new requests wait in the pool queue.

This waiting is not always bad.

It can be healthy.

It means the application is saying:

The database is already handling the maximum number of concurrent operations we allow.

Do not send more work yet.

Without this limit, the application might open too many connections or send too many concurrent queries.

That can make the database slower for everyone.

A well-sized pool keeps the database busy but not overwhelmed.

22. Common Pool Settings

Most connection pools expose settings like these.

22.1. Maximum Pool Size

The maximum number of connections the pool can keep open.

Example: 

max pool size = 10

At most 10 connections can be active.

This controls database concurrency from this application instance.

22.2. Minimum Pool Size

The number of connections the pool tries to keep ready.

Example: 

min pool size = 2

This avoids creating all connections during a sudden traffic spike.

22.3. Acquire Timeout

How long a request waits for a connection before failing.

Example: 

acquire timeout = 5 seconds

If no connection becomes available within 5 seconds, the request gets an error.

This prevents requests from waiting forever.

22.4. Idle Timeout

How long an unused connection can stay open before the pool closes it.

Example: 

idle timeout = 30 seconds

This prevents unnecessary idle connections from staying open forever.

22.5. Max Lifetime

How long a connection can live before being recycled.

This helps avoid stale connections, load balancer timeouts, server-side connection resets, and long-lived broken state.

23. Failure Cases

Connection pools also need to handle failures.

Examples:

Database connection closed unexpectedly.

Query timed out.

Request waiting in queue timed out.

Connection became unhealthy.

Database restarted.

Network partition happened.

Authentication expired.

Transaction was left open by mistake.

When a connection fails, the pool may:

  1. Remove it from the pool.
  2. Reject the current query.
  3. Roll back any transaction if possible.
  4. Create a replacement connection.
  5. Wake up waiting requests if capacity becomes available.
  6. Fail waiting requests if the pool cannot recover.

This is another reason pool implementation is more complex than it looks.

24. One Important Limitation

A single database connection usually cannot safely execute unlimited queries at the same time.

Many database protocols expect one active query at a time per connection, or they have strict rules for pipelining/multiplexing.

So if you have: 

pool size = 10

That often means:

At most 10 database queries can actively use connections at the same time from this application instance.

Extra requests wait in the pool queue.

Some systems support multiplexing, but you should not assume it unless your driver and database protocol explicitly support it.

25. Summary

Overall, the event loop is an amazing example of how we can write concurrent code that can handle multiple things at once without blocking unnecessarily.

In this article, I mainly discussed what the event loop is and how it works with connection pools. We saw how a single event loop can coordinate multiple connections by reacting to events like socket readiness, query completion, timeouts, and released connections.

But event loops are not limited to connection pools. They are used in many different places, such as:

  1. Web servers
  2. Browsers
  3. Database clients
  4. Chat applications
  5. WebSocket servers
  6. AI agents
  7. Game engines
  8. GUI applications
  9. Message queue consumers
  10. IoT systems

The common idea is always the same:

Thanks for reading.
